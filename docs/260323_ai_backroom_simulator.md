# [DEV] AI Backroom Simulator (생성형 백룸 시뮬레이터 확장 기획)

- **대상 팀**: `[Design/Development/Visual]`
- **우선순위**: `[Medium]`
- **상태**: `TODO`

## 1. 개요

현재 `OPEN -> CLOSING -> 다음 날` 루프는 조리와 정산의 밀도는 높지만, "가게 뒤편에서 하루의 흔적이 어떻게 축적되는가"를 보여주는 지속 공간이 부족하다.

본 기획은 영업 후 진입 가능한 **Backroom Simulator**를 도입하여, 주방 뒤편 공간이 날이 갈수록 계속 증식하고, 그 안을 worker들이 돌아다니며 물건을 옮기고 정리하고 쌓아 올리는 경험을 제공하는 것을 목표로 한다.

핵심 아이디어는 다음과 같다.

- **구조(Structure)는 규칙 기반으로 설계**한다.
- **디테일(Decor)은 생성형 AI 기반 대량 에셋 생산으로 확장**한다.
- **의미(Meaning)는 worker 시뮬레이션과 운영 결과 누적으로 만든다.**

즉, AI가 방 전체를 즉흥적으로 "그려내는" 것이 아니라, 사람이 설계한 구조와 규칙 위에 AI가 **롱테일 장식 에셋**을 공급하고, 시뮬레이션이 그 공간에 기억과 서사를 남기게 한다.

## 2. 기획 목표

### 2.1 플레이어 경험 목표

- [ ] 플레이어는 Backroom을 통해 "가게가 실제로 살아 있고 커지고 있다"는 감각을 받아야 한다.
- [ ] Backroom은 단순 배경이 아니라, 영업 결과와 메타 성장의 흔적이 남는 **지속형 공간**으로 인식되어야 한다.
- [ ] 공간 증식은 랜덤 맵 생성보다 "운영의 부산물이 축적된다"는 느낌을 우선해야 한다.
- [ ] worker의 이동과 작업은 플레이어가 개입하지 않아도 관찰 재미를 제공해야 한다.
- [ ] 하루가 지나면 Backroom의 일부가 눈에 띄게 달라져야 한다.

### 2.2 시스템 목표

- [ ] 기존 `Backroom 페이즈` 아이디어를 단발성 정리 단계가 아니라 **영구 성장형 시뮬레이션 레이어**로 재정의한다.
- [ ] 조리/서빙 중심의 코어 루프를 해치지 않도록 Backroom은 짧고 반복 가능한 길이로 유지한다.
- [ ] 룸 생성은 무한 확장 가능하되, 탐색과 가독성을 해치지 않는 규칙을 가져야 한다.
- [ ] 생성형 AI는 핵심 플레이 가독성을 해치지 않는 범위에서만 사용한다.

## 3. 디자인 테제 (Creative Thesis)

### 3.1 이 기획이 좋은 이유

- [ ] 라멘집 운영이라는 테마에 "보이지 않는 노동의 세계"를 추가한다.
- [ ] 플레이어가 만든 성공/실패가 수치가 아니라 물리적 공간 변화로 남는다.
- [ ] 대량의 장식물, 포스터, 박스, 저장품, 메모, 낙서, 잡동사니를 AI로 확장하면 사람이 일일이 만들기 어려운 밀도를 확보할 수 있다.
- [ ] worker의 반복 이동은 공간을 단순한 배경이 아니라 **작동 중인 생태계**로 보이게 만든다.

### 3.2 반드시 피해야 할 방향

- [ ] 런타임에서 방 전체를 자유 생성해 충돌, 동선, 시인성을 불안정하게 만드는 설계는 피한다.
- [ ] 생성형 결과물이 게임 플레이보다 먼저 보이는 "슬롭성 랜덤 배경"으로 흐르면 안 된다.
- [ ] Backroom이 메인 조리 루프를 압도하거나, 별도 게임처럼 지나치게 비대해지면 안 된다.

## 4. 핵심 Pillar

### 4.1 Persistent Accretion

- [ ] 공간은 매일 새로 리셋되지 않고 누적 성장해야 한다.
- [ ] 새 방, 새 상자, 새 장식, 새 고장난 물건, 새 메모가 이전 상태 위에 덧붙는 구조여야 한다.

### 4.2 Readable Chaos

- [ ] 화면은 어수선하고 풍부해 보이되, 중요한 길, 문, 상호작용 지점, worker 동선은 항상 읽혀야 한다.
- [ ] 혼란은 미감으로 쓰고, 플레이 정보는 명확히 분리한다.

### 4.3 Ambient Labor

- [ ] worker는 단순 장식이 아니라 실제 작업 상태를 반영한 행동을 보여야 한다.
- [ ] 물건을 옮기고, 쌓고, 청소하고, 쉬고, 다시 움직이는 흐름이 관찰 가능해야 한다.

### 4.4 Meaningful Memory

- [ ] 공간에는 운영 결과가 축적되어야 한다.
- [ ] 예: 적자가 길어지면 임시 보관 박스와 미정리 물건이 늘고, 흑자가 이어지면 정리된 선반과 개선된 작업 공간이 생긴다.

### 4.5 Endless Curiosity

- [ ] 플레이어는 "다음 날 문 하나가 더 열릴 것 같은 기대"를 가져야 한다.
- [ ] 그러나 무한 확장은 완전 랜덤이 아니라, 운영 상태와 선택 결과에 의해 설득력 있게 열려야 한다.

## 5. 게임 내 정의

### 5.1 Backroom의 역할

- [ ] Backroom은 영업 뒤편의 저장/정비/휴식/폐기/준비 공간이다.
- [ ] Backroom은 `OPEN`의 결과를 흡수하고 `다음 OPEN`의 조건을 준비하는 메타 허브다.
- [ ] Backroom은 플레이어가 직접 꾸미는 하우징이 아니라, **운영과 시뮬레이션으로 자라나는 공간**을 우선한다.

### 5.2 Worker의 역할

- [ ] worker는 Backroom의 상태 변화를 시각화하는 핵심 존재다.
- [ ] worker는 최소한 `운반`, `정리`, `휴식`, `대기`, `작업 실패/지연` 상태를 가진다.
- [ ] worker 수나 효율은 성장 요소가 될 수 있으나, MVP에서는 관찰 가능한 기본 루프가 우선이다.

### 5.3 공간 성장의 의미

- [ ] 방이 늘어나는 이유는 단순 랜덤이 아니라 실제 운영 수요 증가여야 한다.
- [ ] 예: 재고가 늘어 저장실 확장, 유명세가 올라 굿즈 박스 유입, 특수 식재료 해금으로 냉장 공간 확장, 잡다한 이벤트로 임시 창고 증설.

## 6. 생성형 AI 사용 원칙

### 6.1 In Scope

- [ ] AI는 다음과 같은 **롱테일 장식 에셋** 생산에 사용 가능하다.
  - 포스터, 안내문, 박스 라벨, 영수증 더미, 창고 표식, 장비 스티커
  - 선반 위 잡동사니, 벽 얼룩, 정리 바구니, 폐기품 묶음, 작업 메모
  - 테마별 소도구 세트 (기본 창고, 냉장실, 휴게실, 청소 구역, 수리 코너)
- [ ] AI 결과물은 사람이 큐레이션한 스타일 가이드와 팔레트 제약을 따라야 한다.
- [ ] 생성 에셋은 게임 내에서 재조합 가능한 모듈로 정규화되어야 한다.

### 6.2 Out of Scope

- [ ] 출시 빌드에서 텍스트-투-이미지 런타임 생성 기능은 필수 요구사항이 아니다.
- [ ] AI가 타일 충돌, 문 배치, 핵심 동선, 인터랙션 가능 오브젝트를 직접 결정하면 안 된다.
- [ ] worker 내비게이션이 필요한 지형은 생성형 결과에 의존하면 안 된다.

### 6.3 품질 가드레일

- [ ] 장식 에셋은 공용 팔레트, 명암 단계, 윤곽선 밀도, 시점 규칙을 준수해야 한다.
- [ ] 스타일이 크게 어긋나는 에셋은 자동 제외 또는 수동 검수 대상으로 분류되어야 한다.
- [ ] 같은 룸 내 장식은 랜덤이더라도 테마 일관성을 유지해야 한다.

## 7. 공간 생성 원칙

### 7.1 구조 생성

- [ ] 방의 골격은 사람이 설계한 룸 archetype과 연결 규칙으로 생성되어야 한다.
- [ ] 각 룸은 명확한 기능 태그를 가진다.
  - 예: `storage`, `cold_storage`, `staff_rest`, `cleanup`, `repair`, `overflow`, `archive`
- [ ] 룸 간 연결은 최소한의 가독성 규칙을 따른다.
  - 예: 주동선 폭 보장, dead-end 남발 금지, 문 위치 규칙화, 작업 지점 근처 여유 공간 확보

### 7.2 장식 생성

- [ ] 장식은 룸 기능, 최근 운영 상태, 누적 이벤트를 입력으로 배치되어야 한다.
- [ ] 같은 storage라도 "정리형", "과적형", "임시형", "이벤트 적치형" 같은 하위 분위기 변형이 있어야 한다.
- [ ] 장식은 예쁜 배경이 아니라 상태 표현 수단이어야 한다.

### 7.3 증식 규칙

- [ ] Backroom은 매일 무조건 커지지 않는다.
- [ ] 확장은 특정 조건 충족 시 발생해야 한다.
  - 예: 연속 흑자, 재고량 임계치 초과, 업그레이드 구매, 특수 이벤트 발생, worker 부족
- [ ] 확장 시 플레이어는 "왜 이 공간이 생겼는지" 이해할 수 있어야 한다.

## 8. worker 시뮬레이션 요구사항

### 8.1 관찰 가능한 행동

- [ ] worker는 최소한 박스를 들고 이동하거나, 선반을 정리하거나, 청소 장비를 끌고 이동하는 행동을 보여야 한다.
- [ ] 행동은 실제 룸 기능과 연결되어야 하며, 무의미한 배회처럼 보이면 안 된다.
- [ ] worker의 이동은 Backroom이 계속 작동 중이라는 인상을 줘야 한다.

### 8.2 상태 기반 반응

- [ ] Backroom 상태가 나빠질수록 worker는 바빠 보이거나 비효율적으로 움직여야 한다.
- [ ] Backroom 상태가 안정적일수록 worker는 더 정돈된 루틴을 보여야 한다.
- [ ] 플레이어 선택은 worker의 작업 우선순위에 반영되어야 한다.

### 8.3 서사적 흔적

- [ ] worker 행동 결과는 공간에 남아야 한다.
- [ ] 예: 옮겨진 상자, 비워진 선반, 방치된 카트, 정리된 구역, 고장 테이프가 붙은 장비

## 9. 플레이 루프 연결

### 9.1 일일 루프

- [ ] 영업 종료 후 Backroom 진입은 자연스러운 하루 마감 감정을 제공해야 한다.
- [ ] 플레이어는 Backroom에서 짧게 현재 상태를 확인하고, 다음 날을 위한 우선순위를 정해야 한다.
- [ ] Backroom 결과는 다음 영업의 체감 가능한 조건 변화로 이어져야 한다.

### 9.2 플레이어 개입 방식

- [ ] MVP에서 플레이어 개입은 **완전한 직접 배치**보다 **우선순위 지정과 선택지 기반 개입**을 우선한다.
- [ ] 플레이어는 하루당 1~3개의 운영 지시를 내릴 수 있어야 한다.
  - 예: `재고 정리 우선`, `청소 우선`, `냉장실 확장 준비`, `폐기물 정리`, `휴게실 정비`
- [ ] 개입 결과는 다음날 시각 변화와 시스템 변화로 확인 가능해야 한다.

### 9.3 코어 루프와의 관계

- [ ] Backroom은 조리 루프의 보상/회복/준비 레이어로 기능해야 한다.
- [ ] Backroom이 너무 길어지면 반복 피로가 발생하므로, 조작보다 관찰과 선택을 우선한다.
- [ ] 조리에 집중하고 싶은 플레이어를 위해 간소화 또는 자동 처리 옵션을 제공할 수 있어야 한다.

## 10. MVP 범위

### 10.1 MVP에서 반드시 포함할 것

- [ ] 지속 저장되는 단일 Backroom 레이아웃
- [ ] 최소 4종의 룸 archetype
  - `storage`, `cleanup`, `staff_rest`, `overflow`
- [ ] 최소 2~3종의 worker 행동 루프
  - 운반, 정리, 휴식
- [ ] 운영 상태에 따라 달라지는 장식 밀도/정돈도 표현
- [ ] 하루 1회 우선순위 선택
- [ ] 방 또는 구역이 점진적으로 해금되는 성장 감각

### 10.2 MVP에서 제외할 것

- [ ] 자유 인테리어 배치 시스템
- [ ] 정교한 worker 개별 스탯/관계 시뮬레이션
- [ ] 거대한 미로형 무한 탐색 구조
- [ ] 런타임 생성형 이미지 호출
- [ ] 플레이어가 Backroom 안에서 장시간 머무는 별도 장르화

## 11. 성공 기준

- [ ] 플레이어는 3일 이상 진행 시 Backroom이 "살아 있는 공간"으로 느껴졌다고 인지해야 한다.
- [ ] 플레이어는 Backroom 변화를 보고 전날의 운영 결과를 추론할 수 있어야 한다.
- [ ] 반복 관찰 시 worker 행동이 완전히 장식적이라고 느껴지지 않아야 한다.
- [ ] 룸이 늘어나도 주요 동선과 시각 정보는 혼잡해지지 않아야 한다.
- [ ] AI 사용 여부를 몰라도 일관된 아트 방향과 세계관 밀도가 유지되어야 한다.

## 12. 리스크 및 대응 방향

### 12.1 스타일 붕괴

- [ ] 생성형 에셋이 섞이며 통일감이 무너질 수 있다.
- [ ] 대응 방향: 공용 팔레트, 룸별 테마 팩, 수동 큐레이션, 실패 에셋 제외 규칙 필요

### 12.2 공간 가독성 저하

- [ ] clutter가 늘어날수록 이동 경로와 상호작용 정보가 묻힐 수 있다.
- [ ] 대응 방향: 구조층과 장식층 분리, 동선 보호 규칙, 밀도 상한 필요

### 12.3 의미 없는 랜덤성

- [ ] 랜덤 장식이 늘어도 플레이 의미가 없으면 빠르게 질린다.
- [ ] 대응 방향: 모든 변화는 운영 상태 또는 player choice와 연결되어야 한다.

### 12.4 스코프 폭주

- [ ] Backroom이 별도 게임처럼 커지면 현재 코어 루프 개발을 압박할 수 있다.
- [ ] 대응 방향: MVP는 `짧은 관찰 + 하루 1회 선택 + 누적 변화`까지만 고정

## 13. 오픈 질문

- [ ] Backroom의 시점은 기존 게임의 시각 언어와 얼마나 연속적이어야 하는가.
- [ ] worker의 정체는 직원, 로봇, 요정, 고양이 스태프 중 무엇이 세계관상 가장 설득력 있는가.
- [ ] Backroom 성장 결과를 오직 시각 변화로 보여줄지, 명시적 수치(`organization`, `hygiene`, `capacity`)도 병행할지 결정이 필요하다.
- [ ] 공간 확장을 "자동 발생"으로 둘지, "플레이어 승인형"으로 둘지 결정이 필요하다.
- [ ] 이 시스템을 `Backroom 페이즈`로 유지할지, `항상 존재하는 배경 시뮬레이션`으로 확장할지 결정이 필요하다.

## 14. 주요 작업 항목

- [ ] 기존 `Backroom 운영 페이즈`와 본 기획의 관계를 정리해 명칭과 범위를 확정한다.
- [ ] 룸 archetype 목록과 각 룸의 기능 태그를 정의한다.
- [ ] worker 기본 행동 세트와 우선순위 시스템을 정의한다.
- [ ] 운영 결과가 Backroom 상태로 변환되는 핵심 입력값을 정리한다.
- [ ] 생성형 AI 에셋 제작 파이프라인의 스타일 가이드와 검수 기준을 정의한다.
- [ ] MVP에 필요한 최소 장식 세트와 룸 확장 이벤트 세트를 정리한다.
- [ ] 플레이어 개입 방식(우선순위 카드, 정책 선택, 자동 처리)의 UX 방향을 확정한다.
